PART 3 — PUBLIC SHARE LINKS
============================================================

Add public share link functionality to Stats Control and GFX Control pages.

Schema:
- New table match_share_links: id (uuid), match_id (FK), 
  page_type ('stats' | 'gfx'), token (random 32-char URL-safe), 
  created_by_user_id (FK), created_at (timestamptz), 
  revoked_at (timestamptz nullable), expires_at (timestamptz).

Behavior:
- Token is generated with crypto.randomBytes — URL-safe, hard to guess.
- expires_at = match.scheduled_end + 6 hours (or match.scheduled_at + 4 
  hours if no end time exists). Hard expiration; not extendable.
- Generating a new link for the same (match_id, page_type) revokes any 
  prior active link by setting its revoked_at = NOW().
- Public URL pattern: /share/stats/:token and /share/gfx/:token (no auth 
  required; the token IS the auth).
- Hitting a public share link grants the holder full edit access to that 
  one match's Stats Control or GFX Control page. Same UI as the admin 
  version. The visitor does NOT see other matches in the game picker — 
  the dropdown is hidden or replaced with the single match name.
- Server-side: every API call from a public-link session must validate 
  the token is active (not expired, not revoked) AND matches the match_id 
  on the resource being modified.
- UI on the admin side: in the share section, show "Active link expires 
  [time]" with [Copy] and [Regenerate] buttons. Regenerate confirms with 
  a "this will invalidate the current link" warning before issuing a new 
  one.
- When a link is expired or revoked, the public page renders a clear 
  "This link has expired or been revoked" state — do not silently fail.

PART 5 — LINK PREVIEWS / OPEN GRAPH METADATA
============================================================

Currently every page on the app shares the same generic metadata when 
linked in iMessage, WhatsApp, Slack, social, etc. Make link previews 
content-aware so they show useful info per page.

Implement Open Graph and Twitter Card meta tags on these spectator routes:

- /tournaments/:id (TournamentDetail) — og:title = tournament name; 
  og:description = "Tournament at [club] · [date range] · [team count] 
  teams"; og:image = tournament logo or generated card with name + dates.
- /match/:id (MatchDetail) — og:title = "[Team A] vs [Team B]"; 
  og:description = "[tournament name] · [scheduled date] · [status — 
  live, scheduled, final score]"; og:image = generated card with team 
  logos and score (or scheduled time).
- /clubs/:id or /clubs/:slug (ClubDetail) — og:title = club name; 
  og:description = "[location] · [active tournament count] tournaments"; 
  og:image = club logo.
- /players/:id (PlayerProfile) — og:title = player name; og:description 
  = "Handicap [N] · [home club] · [season goals] goals this season"; 
  og:image = player headshot or gray placeholder.
- /teams/:id or wherever teams are displayed publicly — og:title = team 
  name; og:description = "Tournament: [tournament name] · [roster names 
  comma separated]"; og:image = team logo.
- / (homepage) — keep current generic preview.

Implementation notes:
- Vite SPAs default to client-side rendering, which means meta tags set 
  via JS aren't seen by social crawlers. Two acceptable approaches:
  (a) SSR meta injection for spectator routes via a small Express 
      middleware on the API server that detects bot user-agents 
      (facebookexternalhit, Twitterbot, Slackbot-LinkExpanding, 
      WhatsApp, Discordbot, LinkedInBot, etc.) and serves a minimal 
      HTML shell with the right meta tags.
  (b) Pre-render meta tags using a route-aware HTML template at build 
      time for known stable routes, fall back to dynamic for entity 
      pages.
- Pick (a). It's simpler to maintain and supports dynamic data lookups 
  for entity pages. Use cheap server-side queries — tournament, match, 
  player lookups by ID from the DB.
- og:image generation: for now, use existing logos/headshots where 
  available, fall back to a static branded image. A future task can add 
  dynamic image generation; out of scope here.
